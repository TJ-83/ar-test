# 3D Dimensions Team Development Guide

## 1. 目标

本指南用于规范团队后续对 `3D Dimensions` 功能的协作方式，减少：

- 参数被随意修改
- 修复一个视角破坏另一个视角
- 新产品复制时出现不一致实现

## 2. 角色建议

推荐最少明确以下角色：

- 功能 Owner
  - 负责架构一致性
  - 负责参数结构设计
- 产品接入开发
  - 负责单个 SKU 接入与调参
- QA / 验收人员
  - 负责跨角度、跨交互方式验证

## 3. 开发原则

### 3.1 样式冻结优先

在没有明确产品设计更新前，不随意改动：

- 虚线样式
- tick 样式
- 标签字体与颜色
- 三轴“共用一个原点”的设计

### 3.2 参数优先，不写特殊逻辑

新产品接入时优先通过配置解决问题：

- 先调 `originWorldOffset*`
- 再调 `w/h/dEndOffset*`
- 最后调 `screenShift*`

不要第一时间新增特殊条件分支。

### 3.3 单次只改一层

排查问题时建议一次只动一层参数：

- 原点层
- 线终点层
- 屏幕微调层

避免同时修改多个层次，导致无法判断真实影响。

## 4. 推荐开发流程

### 4.1 新产品接入

1. 复制当前稳定尺寸方案
2. 先确认模型包围盒是否正确
3. 先调原点位置
4. 再调三条线长度
5. 最后做标签和屏幕微调
6. 做正面、侧面、背面、自动旋转验证

### 4.2 修复 Bug

修尺寸问题时，先明确问题属于哪一类：

- 原点错误
- 单条线长度错误
- 屏幕偏移错误
- 刷新时机错误
- 视角切换边界问题

修复时不建议把不同类别的问题混在一个 PR 中。

## 5. 代码评审清单

涉及尺寸功能的 PR，评审时至少检查：

- 是否继续沿用 `dimensionConfig`
- 是否新增了魔法数字
- 是否破坏了自动旋转刷新
- 是否影响了 `camera-change` 路径
- 是否改变了三轴共用原点的结构
- 是否补充了文档说明

## 6. 文档更新要求

以下情况必须同步更新文档：

- 配置结构变化
- 核心绘制逻辑变化
- 原点选择逻辑变化
- 刷新机制变化
- 新增已知限制或取消旧限制

## 7. 分支与提交建议

建议团队采用：

- 一个功能分支只解决一个清晰目标
- 提交信息明确包含功能名

推荐提交信息示例：

- `Add dimension config presets for multi-product rollout`
- `Fix auto-rotate redraw for dimensions`
- `Refine dimension endpoint offsets for Somnus cabinet`

## 8. 建议抽离方向

当产品数量增加后，推荐把以下内容从 `cabinet-viewer.html` 中抽离：

- 尺寸配置对象
- 尺寸初始化逻辑
- 尺寸绘制逻辑
- 自动刷新循环

目标结构建议：

- `js/dimensions/config.js`
- `js/dimensions/runtime.js`
- `js/dimensions/presets.js`

## 9. 不建议的团队行为

以下行为会显著提高后续维护成本：

- 每个产品复制一份不同实现
- 直接在页面里继续堆临时 if/else
- 不记录参数意义
- 不验证自动旋转场景
- 不验证背面视角就提交
