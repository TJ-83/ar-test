# 3D 尺寸标注开发报告

## 1. 背景与目标

本次开发的目标，是在 `model-viewer` 驱动的 Web3D 柜体预览页中，补齐一套可用的尺寸标注功能，包括：

- 显示 `W / H / D` 三条尺寸线
- 在模型周围显示对应尺寸文字
- 尺寸线在手动旋转和自动旋转时实时跟随
- 尺寸位置可以通过少量参数手动微调

开发主文件为：

- `C:\Users\tao.jiang\Documents\3D Web Project\ar-test\cabinet-viewer.html`

## 2. 初始问题

开发开始时，尺寸功能虽然已经有一部分结构，但整体存在以下问题：

- 标签和线条使用了两套不同的定位逻辑
- 一部分代码依赖 `Three.js` 相机投影，一部分代码依赖 `model-viewer` 的 hotspot
- 尺寸线能显示，但位置不稳定，和模型不够对齐
- 自动旋转时尺寸线不会自动刷新
- 代码中存在一版旧的 Three.js 投影实现，与新版实现并存，增加了理解成本

最核心的问题是：标签、线条、相机更新并不在同一套坐标体系和刷新节奏里。

## 3. 方案选择与权衡

开发过程中讨论过三种方案。

### 3.1 纯 Three.js 投影

优点：

- 理论上几何关系完整
- 可以直接按 3D 点投影到屏幕

缺点：

- 依赖 `model-viewer` 内部 Three.js 结构，耦合高
- 容易出现坐标空间不一致
- 调试成本高，稳定性差

### 3.2 纯 hotspot + DOM/SVG

优点：

- 与 `model-viewer` 自身机制一致
- DOM 坐标读取简单直观
- 不再依赖内部 Three.js 细节

缺点：

- 只靠 3D 坐标难以完成细腻的视觉微调

### 3.3 最终采用：混合方案

最终保留的方案是：

- 用 3D hotspot 决定三轴几何关系
- 用屏幕空间 SVG 绘制尺寸线
- 用屏幕空间偏移做最终视觉微调

这套方案的特点是：

- 几何关系仍然来自模型本身
- 视觉位置可以手动微调
- 实现复杂度适中
- 后续维护成本较低

## 4. 最终实现结构

### 4.1 热点与绘制职责拆分

在最终版本里，职责划分如下：

- `setupDimensions()`
  - 读取模型包围盒尺寸与中心点
  - 计算三轴公共原点和 `W/H/D` 三个终点
  - 更新 6 个 anchor hotspot 的 `data-position`
- `drawDimensionLines()`
  - 读取 anchor 的屏幕坐标
  - 生成 SVG 线段与 tick
  - 计算文字标签的屏幕位置

### 4.2 三轴公共原点

最终保留了“三条尺寸线共用一个原点”的样式，这更接近坐标轴的视觉表达。

三轴交点本身由 3D 模型角点决定，再经过屏幕偏移做收尾调整。

### 4.3 自动旋转刷新

手动旋转时，尺寸线仍通过 `camera-change` 重绘。

自动旋转时，补充了一条轻量级的 `requestAnimationFrame` 刷新循环，只有在以下条件下才会运行：

- 尺寸显示开启
- 自动旋转开启

关闭尺寸或关闭自动旋转后会立即停止循环，避免不必要的持续重绘。

## 5. 关键改动点

### 5.1 增加 anchor hotspot 机制

页面中保留并启用了 6 个 anchor hotspot：

- `dimAnchorWa`
- `dimAnchorWb`
- `dimAnchorHa`
- `dimAnchorHb`
- `dimAnchorDa`
- `dimAnchorDb`

它们用于提供三轴端点的屏幕投影位置。

### 5.2 修复 hotspot 更新方式

为保证 `model-viewer` 正确接收更新，最终采用：

- 同时写入元素的 `data-position`
- 同时调用 `viewer.updateHotspot({ name, position })`

并修复了 `name` 必须使用完整 slot 名的问题。

### 5.3 标签改为独立 overlay 定位

最终 `W / H / D` 三个标签不再作为 hotspot slot 使用，而是：

- 保留普通 DOM 元素
- 在 `drawDimensionLines()` 中直接设置 `left/top`

这样可以避免标签被 hotspot 跟踪机制干扰，也更适合视觉微调。

### 5.4 建立统一调参入口

最终把核心调参项集中到：

- `dimensionConfig`

当前包含：

- `originWorldOffsetX/Y/Z`
- `wEndOffsetX/Y/Z`
- `hEndOffsetX/Y/Z`
- `dEndOffsetX/Y/Z`
- `screenShiftX/Y`
- `labelOffset`

这样后续可以分层调试：

- 先调原点
- 再调每条线的终点
- 最后调屏幕整体位置与文字位置

## 6. 当前保留的方案特征

目前保留的是一套“以稳定可调为优先”的方案，主要特点如下。

### 6.1 优点

- 功能已经可用
- 三轴样式统一
- 手动旋转与自动旋转都能正确跟随
- 参数化程度已经足够支持继续迭代
- 适合在当前单页面结构中快速维护

### 6.2 已知限制

当前方案仍然存在一个设计上可接受但客观存在的局限：

- 当前视角在模型正面时，线轴与模型边界的贴合感更强
- 当视角旋转到背面时，线轴会显得比正面更“离开模型”

原因不是刷新错误，而是当前方案本质上是：

- 固定 3D 角点
- 叠加固定屏幕偏移

这种方式在正面角度通常更自然，但在背面角度不会自动切换到“当前视角下最贴边的那组角点”。

如果未来要继续优化这一点，建议的方向是：

- 根据相机朝向自动切换三轴原点所绑定的模型角点

但这属于下一阶段增强，不属于当前交付范围。

## 7. 当前建议的使用方式

若后续需要继续手动调节，建议按以下顺序：

### 7.1 调三轴交点

优先调：

- `originWorldOffsetX`
- `originWorldOffsetY`
- `originWorldOffsetZ`

### 7.2 调每条线长度与贴边程度

分别调：

- `wEndOffset*`
- `hEndOffset*`
- `dEndOffset*`

### 7.3 调屏幕位置与文字

最后调：

- `screenShiftX`
- `screenShiftY`
- `labelOffset`

这种顺序最不容易“调一项坏三项”。

- 功能完整度足够

2. 可见性/遮挡优化
